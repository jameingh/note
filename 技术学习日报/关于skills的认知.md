### 一、AgentSkill 是什么
**AgentSkill（简称 Skill）** 是 AI 智能体（Agent）的**模块化、可复用、可移植的专业能力包**。
- 本质：**标准化文件夹**，核心是 `skill.md`（YAML+Markdown）+ 可选脚本、参考资料、模板。
- 作用：把人类的 SOP、领域知识、工作流、工具用法，写成 AI 能读懂并执行的**结构化指令**。
- 类比：给 AI 装**专业插件/技能卡**，让通用大模型变成领域专家。

典型结构：
```
my-skill/
├─ skill.md       # 核心：名称、描述、触发、步骤、格式
├─ references/    # 参考文档、政策、模板
├─ scripts/       # Python/Shell 等可执行脚本
└─ assets/        # 示例、Schema、素材
```

### 二、AI 为什么会按 Skill 执行
1. **上下文注入（核心机制）**
   - 启动时：只加载所有 Skill 的**元数据**（name/description/trigger），轻量、不占 Token。
   - 触发后：把完整 `skill.md` 注入当前对话上下文，**强制约束 AI 行为**。
   - 执行中：按需加载参考资料、脚本、模板，按步骤执行。

2. **软指令 + 推理约束**
   - Skill 不是硬代码，而是**强提示词规范**：告诉 AI“**必须按这个流程、格式、规则来**”。
   - 模型用 Chain-of-Thought 匹配任务 → 匹配 Skill → 加载 → 执行。

3. **工程化保障**
   - 平台（Claude/Cursor/Copilot/OpenClaw）内置 **Skill 调度器（Router）**。
   - 系统提示明确：“**你有这些技能，匹配时必须调用并遵循**”。

### 三、为什么有时不按 Skill 执行（常见原因）
#### 1. 触发失败（最常见）
- **描述/触发词模糊**：`description` 写得太泛，AI 匹配不到。
- **语义不匹配**：用户输入和 Skill 描述**偏差大**。
- **隐式触发依赖模型理解**：模型**误判意图**，觉得“不用 Skill 也能答”。
- **格式错误**：`skill.md` YAML 头缺失/语法错，AI 读不出元数据。

#### 2. 配置/环境问题
- **路径错误**：Skill 没放指定目录、`skill_path` 配置错。
- **平台/模型不支持**：老模型不支持 Skill 或 ToolCalling。
- **权限/依赖缺失**：脚本缺库、API 密钥无效、限流、超时。
- **审批/开关**：Skill 设为“需人工确认”，未通过。

#### 3. Skill 本身问题
- **指令矛盾/模糊**：步骤不清、前后冲突、格式定义弱。
- **资源缺失**：引用的参考文件/脚本不存在、路径错。
- **冲突**：多个 Skill 覆盖同一任务，AI 选了别的。

#### 4. 模型行为（“不听话”）
- **原生偏好**：模型倾向**自己生成**，不愿加载外部 Skill。
- **上下文干扰**：历史对话太长、噪声多，覆盖 Skill 指令。
- **能力不足**：小模型/弱模型**读不懂复杂 Skill**。

### 四、触发机制（两种）
#### 1. 显式触发（强制）
- 用户/UI **直接指定**：`/use-skill 代码评审`、按钮选择。
- 命令/API 显式调用：`load_skill("pdf-extractor")`。
- 特点：**100% 触发**，不依赖语义匹配。

#### 2. 隐式触发（自动）
- AI 启动：加载所有 Skill 的 `name+description`。
- 用户提问 → 模型**语义匹配** → 判定“该用这个 Skill”。
- 加载完整 `skill.md` → 按步骤执行。
- 关键：**`description` 就是触发路由规则**。

### 五、一句话总结
- **Skill = AI 的专业 SOP 插件**。
- **按 Skill 执行**：因为上下文被注入强指令 + 平台调度。
- **不执行**：多是触发匹配失败、配置错、指令模糊、模型偏好。
- **触发**：显式点名 / 隐式语义匹配 `description`。
