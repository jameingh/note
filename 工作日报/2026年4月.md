问题原因：
策略默认值是enable，用户在更新其他字段时，enable 被意外重置为 true，造成策略被错误地启用。

问题分析：
如果保留默认值，会导致用户在更新其他字段时，enable 被意外重置为 true，造成策略被错误地启用。

## ipsec高性能版本修复子线程查询数据一致性问题

问题原因：
在子线程中查询主线程未提交事务的数据，会导致数据不一致。

解决方法：
修复子线程查询数据一致性问题，先更新子网启用/不启用状态，再查询出子网列表，传入子线程。保证子线程获取的子网数据和主线程一致

## ✅OPT的个人理解

OPT（one person team）行业/领域解决方案。指单个人独立承担产品从 0 到 1 的全链路职责（需求、设计、开发、测试、部署、运营、增长等），以最小组织单元交付完整价值。</br>

预编排的「多 Agent 协作工作流 + 行业 SOP + 数据 + 权限体系」

Agent = LLM + Harness（工具集合）

- 模型决定了 Agent 的能力上限，harness 决定了 Agent 能发挥出多少。
- 光有反馈没有前馈，Agent 会重复犯同样的错；光有前馈没有反馈，规则永远不知道有没有用。两者缺一不可。

### 一般工作模式
team-plan → team-prd → team-exec → team-verify → team-fix

> 不管是《人月神话》还是《大教堂与集市》，都在强调一个观点：软件开发是一个复杂的过程，需要一个 leader 来协调和管理。人与人之间的沟通协作，存在沟通成本。

### 好处：
1、无协作开销，不需要依赖团队成员，自己就能完成工作</br>
2、用户直达，直接对接真实用户，用反馈驱动迭代，避免信息噪音</br>

###坏处：
#### 1、缺乏协作与碰撞 → 完全没变，甚至更孤独
- 个体认知局限性：单人决策容易固执，更容易陷入自我闭环、认知盲区
#### 2、深度受限->反而更容易 “样样松”
- LLM 会让你快速出活，但专业深度还是靠人。
- 架构、安全、合规、复杂业务逻辑、高并发优化……AI 只能辅助，不能替代深度判断。
- 很容易出现：表面全栈，底层虚浮。
#### 3、规模瓶颈->依然存在，只是阈值提高了一点
- LLM 能把单人产能从 ×1 拉到 ×5～×10，但拉不到 ×100。
- 业务一旦到多项目并行、多客户、多版本、强协作阶段，单人还是卡死。
- 本质：人是瓶颈，不是工具。
#### 4、效率假象->更明显、更迷惑人
- LLM 让你每天产出巨多，感觉超高效。
- 但大量产出是碎片、草稿、半成品，真正高质量交付依然有限。
- 很容易陷入：产出爆炸，价值一般。

### 总结
- 好处：无协作开销，用户直达
- 坏处：缺乏协作与碰撞，深度受限，规模瓶颈，效率假象

- LLM 能让一个人更强，却变不成一个真正的团队。
- 能大幅提升上限、降低体力成本，但不能消除 “单人” 这个核心结构缺陷。它是更强的单人，不是一个团队的替代品。
